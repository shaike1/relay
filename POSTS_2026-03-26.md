## LinkedIn Post

**Title:**
I read the viral post about replacing OpenClaw with Claude Channels. Here's what it missed.

---

**Body:**

A detailed write-up has been circulating about replacing OpenClaw with Claude Code's built-in Channels feature, using SyncThing for file sync and a macOS LaunchAgent for crash recovery.

It's a well-written guide. But it describes a problem that already has a more complete open-source solution — one that doesn't require SyncThing, doesn't require a LaunchAgent, and actually has the message queue that Channels lacks.

**The problem the article correctly identifies:**

> "Claude Code Channels has no message queue. If Claude's session isn't running when you send a message, that message is gone. Not saved for later. Not waiting in line. Just lost."

This is real. And it's exactly why we built Relay differently.

**What Relay does:**

Every incoming Telegram message is written to a queue file on disk (`/tmp/tg-queue-{THREAD_ID}.jsonl`) before Claude ever sees it. If Claude is mid-restart, the message waits. When Claude comes back up, the MCP server resumes tailing the file and delivers everything in order. Nothing is lost.

**The architecture:**

```
Phone
  ↓
Telegram topic (one per project)
  ↓
Relay bot (single long-poll)
  ↓
/tmp/tg-queue-{THREAD_ID}.jsonl
  ↓
MCP server (tails queue file)
  ↓
Claude Code (in project directory)
  ↓
send_message → Telegram topic → Phone
```

One bot token. Multiple projects. Each project isolated in its own Telegram topic. The queue file decouples Claude's lifecycle from message delivery.

**Why you don't need SyncThing:**

The article recommends SyncThing to keep `~/.claude/` in sync between your laptop and Mac Mini, because Claude needs to run on a machine that stays on 24/7.

Relay runs Claude on your server — not on your laptop. Your phone is the interface. There's nothing to sync: the server has the project files, Claude runs there, you interact from anywhere. Closing your laptop changes nothing.

**Adding a project is one command:**

```
/new /path/to/project
```

Or for a remote server:

```
/new root@your-server /root/myproject
```

Relay creates the Telegram topic, writes `.mcp.json` into the project directory, starts a tmux session, and launches Claude — all automatically. The topic appears in your group and is live in seconds.

**What else Relay handles that the article doesn't mention:**

• Multiple sessions across multiple servers — all in one Telegram group
• Inline keyboard buttons — Claude sends clickable options directly in Telegram
• Auto-login — when OAuth expires, the bot sends you the re-auth link automatically
• Peer messaging — sessions can coordinate with each other
• Primary/backup redundancy with automatic failover

**Setup:**

```bash
git clone https://github.com/shaike1/relay
cd relay && bash install.sh
```

One script. Handles Python deps, Bun, credentials, and the systemd service.

Open source, MIT licensed: **github.com/shaike1/relay**

If you're setting up a "always-on Claude" setup, happy to answer questions in the comments.

## Reddit Post

**Subreddit:** r/ClaudeAI (cross-post to r/selfhosted)

**Title:**
The viral "replace OpenClaw with Claude Channels" post is good — but Claude Channels has no message queue. Here's an open-source setup that does.

---

**Body:**

A detailed guide has been circulating about replacing OpenClaw with Claude Code's built-in Channels, using SyncThing for file sync between machines and a macOS LaunchAgent for auto-restart.

Worth reading if you haven't seen it. But the author correctly identifies a fundamental limitation:

> "Claude Code Channels has no message queue. If Claude's session isn't running when you send a message, that message is gone. Not saved for later. Not waiting in line. Just lost."

That's a real problem. Here's an open-source alternative that solves it: **github.com/shaike1/relay**

---

**How it works (the short version):**

Each project gets its own Telegram topic (forum thread). Claude Code runs in that project's directory with an MCP server listening. When you send a message → Claude gets it as an event → does the work → replies in the same topic.

The key difference: **every message is written to a queue file on disk before Claude sees it.**

```
/tmp/tg-queue-{THREAD_ID}.jsonl
```

If Claude is mid-restart, the file fills up. When Claude comes back, it picks up from where it left off. Nothing lost.

---

**Why you don't need SyncThing:**

The guide recommends SyncThing to keep `~/.claude/` in sync between your laptop and Mac Mini, because you need a machine that stays on 24/7.

With Relay, Claude runs on a server. Your phone is just the interface. There's nothing to sync — no two machines to keep in agreement, no path matching, no "what happens when I close my laptop." The server has the files, Claude runs there, done.

---

**Adding a project:**

```
/new /path/to/project
```

Or on a remote server:

```
/new root@server-ip /root/myproject
```

Relay creates the Telegram topic, writes `.mcp.json`, starts a tmux session, launches Claude. Topic is live in seconds. No config files touched.

---

**Auto-restart without a LaunchAgent:**

The guide creates a custom shell script + macOS LaunchAgent for crash recovery. With Relay:

- Relay itself runs as a systemd service (auto-starts on boot, restarts on crash)
- Claude sessions run in a `while true` loop in tmux
- Server reboots → Relay comes back up → Claude resumes its last conversation (`--continue`)

Nothing to write. Nothing to configure.

---

**Other things Relay handles:**

- **Multiple projects on multiple servers** — all in one Telegram group, each in its own topic
- **Inline buttons** — Claude can send clickable options: `buttons=[["Yes", "No"]]`
- **Auto-login** — when OAuth token expires, bot extracts the re-auth URL and sends it to you
- **Remote sessions over SSH** — `/new root@100.64.0.12 /root/project` provisions everything remotely
- **Per-topic access control** — specify which Telegram users can message which session
- **Primary/backup redundancy** — watchdog script activates backup relay after 45s downtime

---

**Setup:**

```bash
git clone https://github.com/shaike1/relay
cd relay && bash install.sh
```

One script handles everything. Then `/new /path/to/project` in Telegram.

MIT licensed, open source: **github.com/shaike1/relay**

Happy to answer questions about the architecture.

---

*Edit: the "single long-poll" architecture is worth noting — Telegram returns 409 Conflict if two processes call getUpdates with the same bot token. Relay holds the single poll and writes to per-topic queue files. Each MCP server reads only its own file. That's how multiple projects share one bot without conflicts.*
